Section 1

Question 1: What is Git communicating to you?

The git log command shows the commit with an identifier, its author, the timestamp when the commit occurred, and the message I included with the commit.

Question 2: What do you see that is common between this git log command and the previous git commit?

The git commit provides the message I included with the commit and the first few characters of the identifier.

Question 3: Why do you think this is?

I think this would make the git commit easier to track in the git log, especially when the git log becomes very large.

Section 2

Question 1: What do you notice about file1.txt and file2.txt? How are they displayed in git status?

The git status shows that file1.txt had been modified from the previous commit but its changes haven't been staged yet, and file2.txt was untracked by git. The files are displayed in different sections of the git status output.

Question 2: Are they both displayed in the same section? If not, why not?

No, they are in different sections. Since file1.txt had changes previously committed, git at least knew about it and detected that it had updates that were not staged yet. However, file2.txt was just created so git has nothing tracked for this file.

Question 1: What do you notice about file1.txt and file2.txt? How are they displayed in git status?

Both are listed in the same section under "Changes to be commited" in green text.

Question 2: Compare and contrast the manner in which the two files are displayed.

Both are in the same section in green text, but file1.txt shows up as a modified file and file2.txt shows up as a new file.

Section 4

Question 1: How many files are listed as having been changed?

Two. This shows the staged updates for both file1.txt and file2.txt.

Question 2: What are the specific differences listed for each file? In what color is the difference displayed?

The diff shows the lines I added to each file with a plus sign in the left column and in green text.
